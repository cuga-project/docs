Overview

The CUGA Policy System is a sophisticated framework for controlling agent behavior through declarative policies. Policies can guide, block, modify, or format agent actions based on various triggers and conditions. The system integrates seamlessly into the CugaLite graph execution flow, providing fine-grained control over agent behavior.

Key Features

  • Multiple Policy Types: Playbooks, Intent Guards, Tool Guides, Tool Approval, and Output Formatters
  • Flexible Triggers: Keyword, Natural Language, App, State, Tool, and Always triggers
  • Semantic Matching: Uses embeddings and LLM for intelligent policy matching
  • Priority System: Handles conflicts when multiple policies match
  • Vector Storage: Uses Milvus for efficient policy retrieval
  • Context-Aware: Uses PolicyContext to extract relevant information from agent state

PolicyContext: The Bridge Between State and Policies

PolicyContext is a structured data model that serves as the interface between the agent's execution state and the policy matching system. It extracts and normalizes relevant information from the agent state, making it available for policy triggers to evaluate.

When a policy check is initiated, the system first creates a PolicyContext from the current agent state using PolicyConfigurable.create_context_from_state(). This context object is then passed to the policy matching functions, which use it to evaluate triggers and determine if policies should activate.

The PolicyContext provides helper methods like get_target_text() and get_query_text() that intelligently extract the right text for different trigger types, prioritizing the most relevant information (e.g., last chat message over initial user input for better context).

Policy Types

Playbook Step-by-Step Task Guidance

Playbooks provide structured, step-by-step instructions for completing complex tasks. When a playbook matches, its guidance is injected into the agent's prompt, helping the agent follow a predefined process.

  • Purpose: Guide the agent through multi-step processes
  • Action: Injects markdown guidance into the system prompt
  • Refinement: Can be dynamically refined based on user progress
  • Example: "How to process insurance claims" playbook

Intent Guard Blocking and Warnings

Intent Guards intercept user intents before execution and can either block them entirely or provide warnings. They have the highest priority in the matching system.

  • Purpose: Prevent unsafe or unauthorized actions
  • Action: Blocks execution and returns a custom response
  • Override: Can allow user override if configured
  • Example: Blocking deletion of critical data

Tool Guide Tool Description Enhancement

Tool Guides enhance tool descriptions with additional context, examples, or warnings. Multiple tool guides can be applied simultaneously to different tools.

  • Purpose: Provide better context for tool usage
  • Action: Appends or prepends content to tool descriptions
  • Targeting: Can target specific tools or apps
  • Example: Adding usage examples to API tools

Tool Approval Human-in-the-Loop Control

Tool Approval policies require human approval before executing code that uses specific tools. This provides a safety mechanism for sensitive operations.

  • Purpose: Require approval for sensitive tool usage
  • Action: Creates an interrupt to wait for user approval
  • Timing: Checked after code generation, before execution
  • Example: Requiring approval for payment processing tools

Output Formatter Response Formatting

Output Formatters transform the final AI response into specific formats like JSON, structured markdown, or custom templates. They operate on the agent's final output.

  • Purpose: Format agent responses consistently
  • Action: Reformats the final answer using LLM or direct replacement
  • Types: Markdown instructions, JSON schema, or direct string
  • Example: Formatting responses as structured JSON for API integration

Trigger Types

Triggers determine when a policy should be activated. Multiple triggers can be combined, and all must match for a policy to activate (unless using OR logic for keywords).

Keyword Trigger Exact Text Matching

Matches when specific keywords are found in the target text. Supports case-sensitive and case-insensitive matching, with AND/OR logic for multiple keywords.

  • Targets: intent, chat_messages, sub_task, agent_response
  • Operators: AND (all keywords) or OR (any keyword)
  • Use Case: Fast, exact matching for specific terms

Natural Language Trigger Semantic Matching

Uses embedding-based similarity to match user intent semantically. More flexible than keyword matching and can understand variations in phrasing.

  • Targets: intent, chat_messages, sub_task, agent_response
  • Threshold: Similarity score (0.0 to 1.0, default 0.7)
  • Use Case: Understanding user intent beyond exact keywords
  • Multiple Values: Supports multiple NL descriptions (OR logic)

App Trigger Application Context

Matches when a specific application is active in the current context. Useful for app-specific policies.

  • Target: Active apps in the execution context
  • Use Case: App-specific policies and restrictions

State Trigger Agent State Conditions

Matches based on values in the agent's state. Supports equals, contains, and regex operators for flexible state checking.

  • Operators: equals, contains, regex
  • Use Case: Conditional policies based on execution state

Tool Trigger Tool Availability

Matches when specific tools are available or being used. Can trigger before or after tool execution.

  • Stage: before or after tool execution
  • Use Case: Tool-specific policies and monitoring

Always Trigger Unconditional

Always matches, regardless of context. Useful for policies that should always be active.

PolicyContext Structure

PolicyContext is a structured representation of the current execution context, containing all information needed for policy matching:

  • user_input: Current user input/intent
  • chat_messages: Recent chat message history
  • sub_task: Current sub-task being executed
  • agent_response: Last agent response (for Output Formatters)
  • available_tools: List of available tools in current context
  • active_apps: List of active applications
  • state_data: Additional state data for State triggers
  • thread_id: Thread/session identifier
  • metadata: Additional metadata

The PolicyContext is created from the agent state using PolicyConfigurable.create_context_from_state(), which extracts relevant fields and formats them for policy matching.

Matching Algorithm

  1. Context Creation: Extract context from agent state into PolicyContext object
  2. Target Selection: Determine which field to evaluate (intent, agent_response, etc.) using PolicyContext.get_target_text()
  3. Parallel Evaluation: Both keyword and natural language triggers are evaluated independently in parallel against the PolicyContext
  4. Keyword Evaluation: Check all policies with keyword triggers using PolicyContext.get_target_text() (fast exact matching)
  5. Natural Language Evaluation: Use PolicyContext.get_query_text() for vector search to find candidate policies, then LLM conflict resolution if multiple match
  6. Candidate Combination: Combine results from both keyword and NL evaluations into a single candidate list
  7. Priority Resolution: Intent Guards always win over other policy types
  8. Confidence Selection: Among remaining candidates, select highest confidence/priority
  9. Trigger Validation: All non-NL triggers must also match against PolicyContext for a policy to be considered

Priority Rules

  • Intent Guards have the highest priority and always win over Playbooks
  • Within the same type, higher priority values win
  • If priorities are equal, higher confidence scores win
  • Tool Guides are independent and can have multiple matches simultaneously

Why Evaluate Both Keyword and NL?

Both keyword and natural language triggers are evaluated independently and in parallel. This allows:

  • A policy with keyword triggers to match even if it also has NL triggers
  • A policy with only NL triggers to match based on semantic similarity
  • The system to select the best match from all candidates, regardless of trigger type
  • Better coverage - keyword matching is fast and exact, while NL matching handles variations

The results from both evaluations are combined, and the best match is selected based on policy type priority (Intent Guards > others) and confidence scores.

Integration Points

1. prepare_tools_and_apps Node

Policies Checked: Intent Guard, Playbook, Tool Guide
Purpose: Early interception and guidance injection
Actions:

  • Create PolicyContext from current state
  • Check for Intent Guards using PolicyContext (can block immediately)
  • Check for Playbooks using PolicyContext (store guidance in metadata)
  • Check for Tool Guides using PolicyContext (store guide content in metadata)
  • Apply tool guides to tool descriptions

2. call_model Node

Policies Checked: Tool Approval (after code generation)
Purpose: Require approval for sensitive tool usage
Actions:

  • After code generation, create PolicyContext with generated code
  • Check if code uses restricted tools via check_tool_approval_for_code()
  • If approval required, create HITL interrupt
  • Store approval metadata for later processing

3. CugaLiteCallback Node

Policies Checked: Output Formatter
Purpose: Format final responses consistently
Actions:

  • After final answer is generated, create PolicyContext with agent_response field
  • Check for Output Formatters using PolicyContext (target: "agent_response")
  • If matched, reformat response using LLM or direct replacement
  • Replace final_answer with formatted version

Key Flow Characteristics

  • Early Blocking: Intent Guards can block execution before any tools are prepared
  • Guidance Injection: Playbooks inject guidance into prompts, affecting code generation
  • Tool Enhancement: Tool Guides modify tool descriptions before they're shown to the LLM
  • Approval Gates: Tool Approval creates interrupts that pause execution for user confirmation
  • Post-Processing: Output Formatters transform responses after generation but before returning to user

Storage and Retrieval

Policies are stored in Milvus, a vector database that enables efficient semantic search. Each policy is embedded using either OpenAI embeddings or local models, allowing for fast similarity-based retrieval.

Storage Schema

  • id: Unique policy identifier
  • policy_type: Type of policy (playbook, intent_guard, etc.)
  • name: Human-readable name
  • description: Policy description (used for embeddings)
  • embedding: Vector embedding for semantic search
  • policy_json: Full policy definition as JSON
  • priority: Priority value for conflict resolution
  • enabled: Whether the policy is active

Best Practices

Policy Design

  • Use Intent Guards for safety-critical blocking
  • Use Playbooks for complex multi-step processes
  • Use Tool Guides to provide context without changing behavior
  • Use Tool Approval for sensitive operations
  • Use Output Formatters for consistent response formats

Trigger Selection

  • Use Keyword triggers for exact, fast matching
  • Use Natural Language triggers for semantic understanding
  • Combine multiple triggers for more specific matching
  • Set appropriate thresholds for NL triggers (0.7 is a good default)

Priority Management

  • Intent Guards automatically have highest priority
  • Set explicit priority values for policies of the same type
  • Use higher priorities for more specific policies
  • Test policy interactions to avoid conflicts

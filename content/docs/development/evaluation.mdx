# **CUGA Evaluation**

An evaluation framework for **CUGA**, enabling you to **test your APIs** against structured test cases with detailed scoring and reporting.

---

## **Features**
- ✅ Validate **API responses** against expected outputs  
- ✅ Score **keywords**, **tool calls**, and **response similarity**  
- ✅ Generate **JSON** and **CSV** reports for easy analysis  

---

## **Test File Schema**

<Callout type="info" title="Schema">
Your test file must be a **JSON** following the structure below. Expand to view the full JSON schema.
</Callout>

<details>
  <summary><strong>View Full JSON Schema</strong></summary>

```json filename="test_schema.json"
{
  "name": "name for the test suite",
  "title": "TestCases",
  "type": "object",
  "properties": {
    "test_cases": {
      "type": "array",
      "items": {
        "$ref": "#/definitions/TestCase"
      }
    }
  },
  "required": ["test_cases"],
  "definitions": {
    "ToolCall": {
      "type": "object",
      "properties": {
        "name": { "type": "string" },
        "args": { "type": "object" }
      },
      "required": ["name", "arguments"]
    },
    "ExpectedOutput": {
      "type": "object",
      "properties": {
        "response": { "type": "string" },
        "keywords": {
          "type": "array",
          "items": { "type": "string" }
        },
        "tool_calls": {
          "type": "array",
          "items": { "$ref": "#/definitions/ToolCall" }
        }
      },
      "required": ["response", "keywords", "tool_calls"]
    },
    "TestCase": {
      "type": "object",
      "properties": {
        "name": { "type": "string" },
        "description": { "type": "string" },
        "intent": { "type": "string" },
        "expected_output": { "$ref": "#/definitions/ExpectedOutput" }
      },
      "required": ["name", "description", "intent", "expected_output"]
    }
  }
}
```

</details>

<Callout type="warning" title="Schema consistency note">
In the <code>ToolCall</code> definition, the property is named <code>args</code>, but the <code>required</code> array references <code>arguments</code>.  
If this is unintended, change <code>"arguments"</code> to <code>"args"</code> (or rename the property) to ensure validation consistency.
</Callout>

## **Usage**

```bash
cuga evaluate -t <test file path> -r <results file path>
```

**Steps:**

1.  Update <code>mcp\_servers.yaml</code> with your APIs (or create a new YAML), then set:

```bash
export MCP_SERVERS_FILE=<location>
```

2.  Create a test file following the schema.

3.  Run the evaluation command above.

***

<Callout type="success" title="Authoring tips">
• Keep <strong>intents</strong> user-centric and concise.  
• Use <strong>keywords</strong> for must-include facts (IDs, amounts, dates).  
• Specify <strong>tool_calls</strong> only when you expect a deterministic invocation pattern.  
• For free-form responses, choose an appropriate <code>response_scoring_type</code> in your evaluator config.
</Callout>
